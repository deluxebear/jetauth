## 概述

JetAuth 支持多因素认证（Multi-Factor Authentication, MFA），在密码之外增加一层安全验证。系统支持五种 MFA 方式，管理员可在**组织级别**和**用户级别**分别配置 MFA 策略。

---

## 配置层级与优先级

MFA 配置存在两个层级，**用户级优先级高于组织级**：

```
用户级 mfaItems（如果有配置）  →  完全覆盖组织级
         ↓ 如果用户级为空
组织级 mfaItems                →  作为默认策略生效
```

**重要**：用户级一旦有任何 mfaItems 配置（即使只配了一项），组织级的全部配置都不再生效。这是"完全覆盖"而非"合并"。

## 组织级 MFA 配置

### 配置入口

**组织管理** → 选择组织 → 编辑 → **MFA 项目**区域

### 配置项

| 字段 | 说明 |
|------|------|
| MFA 项目 | 配置表格，每行一个 MFA 类型 + 规则 |
| 记住设备时长 | 用户勾选"记住此设备"后免 MFA 验证的时长（小时），默认 12 小时 |

### 操作步骤

1. 进入组织编辑页面
2. 找到 **MFA 项目** 表格
3. 点击 **添加** 按钮
4. 选择 MFA 类型（短信 / 邮箱 / 应用 / 推送）
5. 选择规则（可选 / 提示 / 必需）
6. 可添加多行，配置多种 MFA 类型
7. 使用 ▲▼ 按钮调整优先级顺序
8. 点击 **保存** 生效

### 示例配置

| 名称 | 规则 | 效果 |
|------|------|------|
| 应用 (TOTP) | 必需 | 所有用户登录前必须设置 TOTP |
| 邮箱 | 提示 | 登录后弹通知建议启用，可跳过 |
| 短信 | 可选 | 不提示，用户自行决定 |

## 用户级 MFA 配置

### 配置入口

**用户管理** → 选择用户 → 编辑 → **安全** 标签页 → **多因子认证** 区域

### 两个不同的概念

用户编辑页面有两个 MFA 相关的区域，容易混淆：

#### 1. 多因素认证（启用状态）

显示该用户当前 MFA 的**实际启用状态**，包括：

| 显示内容 | 说明 |
|---------|------|
| 类型名称 | 短信 / 邮箱 / 应用 / Radius / 推送 |
| 已启用 / 已禁用 | 该类型的当前状态 |
| 首选标签 | 标记为用户首选的 MFA 方式 |

**管理员可执行的操作**：

| 操作 | 条件 | 说明 |
|------|------|------|
| 设为首选 | 该 MFA 已启用且非首选 | 设为用户登录时默认选中的 MFA 方式 |
| 启用 | 非 TOTP 类型 + 管理员 + 非自己 | 管理员直接为用户启用短信/邮箱等 MFA |
| 全部禁用 | 任何 MFA 已启用 | 禁用该用户所有 MFA |

**用户自己可执行的操作**：

| 操作 | 说明 |
|------|------|
| 编辑 | 已启用的 MFA，跳转到设置页面重新配置 |
| 设置 | 未启用的 MFA，跳转到设置页面初始配置 |

> **注意**：TOTP（应用）类型不能由管理员为他人启用，因为需要用户本人用认证器 App 扫描 QR 码。

#### 2. MFA 项目（策略配置）

配置该用户的 MFA **策略要求**，和组织级 MFA 项目功能相同但针对单个用户：

- 如果配置了任何项目，**完全覆盖**组织级配置
- 如果为空，则继承组织级配置
- 管理员设置，普通用户没有权限修改

---

## MFA 类型说明

| 类型 | 内部标识 | 说明 | 前置要求 |
|------|---------|------|---------|
| 短信 (SMS) | `sms` | 通过短信发送 6 位验证码 | 用户必须已配置手机号 |
| 邮箱 (Email) | `email` | 通过邮件发送 6 位验证码 | 用户必须已配置邮箱 |
| 应用 (TOTP) | `app` | 认证器应用生成动态验证码 | 用户需安装 Google Authenticator 等应用 |
| 推送 (Push) | `push` | 推送通知验证 | 需配置推送服务提供商 |
| Radius | `radius` | Radius 协议认证 | 需配置 Radius 服务器 |

## 三种规则的行为差异

### 必需（Required）

```
用户登录 → 输入密码 → 后端检查: MFA 是否已设置?
                                    ↓ 未设置
                              返回 RequiredMfa
                                    ↓
                            强制跳转 MFA 设置页面
                            （不完成设置无法登录）
                                    ↓ 已设置
                              返回 NextMfa
                                    ↓
                            跳转 MFA 验证页面
                            （输入验证码完成登录）
```

- 用户**无法跳过**
- 未设置 MFA 时：被拦截在登录流程中，强制完成设置
- 已设置 MFA 时：每次登录需输入验证码（除非勾选了"记住设备"）

### 提示（Prompted）

```
用户登录 → 输入密码 → 登录成功 → 进入系统
                                    ↓
                        右上角弹出通知卡片:
                        "为了账户安全，建议您启用多因素认证"
                        [应用] [邮箱]  ← 显示建议启用的类型
                        [稍后] [去启用]
```

- 用户**可以跳过**（点"稍后"）
- 每次登录都会提示（直到用户启用）
- 不阻断登录流程

### 可选（Optional）

- **不做任何提示**
- 用户可自行到个人设置中启用
- 完全自愿

## 用户设置 MFA 流程

### 触发方式

1. 登录时被 Required 规则强制跳转
2. 登录后点击 Prompted 通知中的"去启用"
3. 用户自行进入个人编辑页面点击"设置"

### 三步设置流程

#### 第一步：验证密码

- 输入当前登录密码
- 验证身份，防止未授权操作
- 仅接受用户自己的密码

#### 第二步：验证码验证

**TOTP（应用）类型**：
1. 页面显示 QR 码和密钥文本
2. 用户使用 Google Authenticator / Microsoft Authenticator 等应用扫描 QR 码
3. 也可手动输入密钥添加账户
4. 输入应用显示的 6 位动态验证码
5. 系统验证码有效性

**短信 / 邮箱类型**：
1. 页面显示目标手机号或邮箱（已脱敏）
2. 点击"发送验证码"按钮
3. 等待验证码（60 秒倒计时后可重发）
4. 输入收到的 6 位验证码
5. 系统验证码有效性

#### 第三步：启用并保存恢复码

1. 系统生成一组恢复码
2. 页面展示恢复码列表
3. 用户必须复制或记录恢复码
4. **恢复码仅显示一次，之后无法再查看**
5. 点击"确认启用"完成设置

## 用户登录 MFA 验证流程

当用户已启用 MFA 后，登录流程如下：

```
输入用户名密码 → 密码验证通过
    ↓
检查是否有未设置的 Required MFA → 是 → 跳转设置页面
    ↓ 否
检查是否有已启用的 MFA → 是 → 跳转验证页面
    ↓ 否
直接登录成功
```

### MFA 验证页面

1. 如果用户有多种 MFA，自动选中**首选方式**
2. 用户可切换其他已启用的 MFA 方式
3. 输入 6 位验证码（或恢复码）
4. 可选勾选"记住此设备"
   - 勾选后，该设备在设定时长内免 MFA 验证
   - 时长由组织的"记住设备时长"配置决定（默认 12 小时）
5. 验证通过后完成登录

## 管理员为用户启用 MFA

### 适用场景

管理员需要为组织内用户批量或个别启用 MFA，无需用户自行操作。

### 操作步骤

1. 进入**用户管理** → 选择目标用户 → 编辑
2. 切换到**安全**标签页
3. 在**多因素认证**区域，找到目标 MFA 类型
4. 点击右侧的**启用**按钮
5. 确认弹窗显示用户信息（用户名、手机号/邮箱）
6. 确认后系统自动为该用户启用 MFA

### 限制

| MFA 类型 | 管理员可启用 | 原因 |
|---------|------------|------|
| 短信 | 可以 | 使用用户已有手机号 |
| 邮箱 | 可以 | 使用用户已有邮箱 |
| 应用 (TOTP) | **不可以** | 需要用户本人扫描 QR 码 |
| Radius | 可以 | 服务端配置 |
| 推送 | 可以 | 服务端配置 |

### 用户级 MFA 策略配置

管理员可以为特定用户配置**独立的 MFA 策略**（覆盖组织级）：

1. 在用户编辑页面找到 **MFA 项目** 表格
2. 添加/修改 MFA 类型和规则
3. 保存后，该用户的 MFA 策略独立于组织配置

**典型场景**：
- 组织级设 TOTP 可选，但对高权限用户设为必需
- 某用户需要豁免组织的 MFA 要求（用户级配空 MFA 项目）

## WebAuthn 生物识别/硬件密钥

### 概述

WebAuthn 允许用户使用指纹、面部识别或硬件安全密钥（如 YubiKey）进行身份验证。

### 注册新凭证

1. 进入用户编辑页面 → 安全标签 → WebAuthn 凭证区域
2. 点击**添加**按钮
3. 浏览器弹出系统验证器 UI（指纹/面部/硬件密钥）
4. 完成验证后凭证自动保存

> **注意**：仅用户本人可添加 WebAuthn 凭证（管理员不能代为添加）

### 删除凭证

在 WebAuthn 凭证列表中，点击对应凭证的**删除**按钮。

## 恢复码

### 用途

当用户丢失认证器设备（手机丢失、更换手机等）时，可使用恢复码代替验证码完成登录。

### 获取方式

恢复码在 **MFA 设置流程的第三步**自动生成，仅显示一次。

### 使用方式

在 MFA 验证页面，选择"使用恢复码"，输入完整的恢复码即可完成登录。

### 注意事项

- 恢复码仅在设置时显示一次，请务必妥善保存
- 建议将恢复码保存在安全的离线位置
- 禁用 MFA 后恢复码同时失效
- 重新设置 MFA 会生成新的恢复码

## 常见问题

### Q: 设置了组织级 MFA 但用户登录时没有触发？

**排查步骤**：
1. 确认组织 MFA 项目的规则是否为"必需"（Required）
2. 检查用户是否有用户级 mfaItems 配置（用户级会完全覆盖组织级）
3. 确认 MFA 类型名称是否正确（必须是小写：`sms`、`email`、`app`、`push`）
4. 如果规则是"提示"（Prompted），MFA 不会阻断登录，只会在登录后弹通知

### Q: TOTP 为什么不能由管理员为他人启用？

TOTP 需要用户本人用认证器 App（Google Authenticator 等）扫描 QR 码获取密钥。这个过程必须在用户的设备上完成，管理员无法代替。

### Q: 用户级 MFA 项目为空和没有配置有什么区别？

- **没有配置（null）**：继承组织级 MFA 配置
- **配置为空数组（[]）**：同样继承组织级（`len(user.MfaItems) == 0`）
- **配置了任何项目**：完全覆盖组织级，即使只有一项

### Q: "记住设备"的作用范围？

- 仅对当前浏览器/设备生效
- 通过 Cookie 记录，清除 Cookie 后失效
- 过期时间由组织的"记住设备时长"配置决定
- 不影响其他设备的 MFA 验证要求

### Q: MFA 设置时密码验证失败？

MFA 设置页面的密码验证只接受**用户自己的密码**，不接受组织万能密码或管理员密码。这是安全设计——确保只有用户本人能设置自己的 MFA。

### Q: WebAuthn 添加时报 "Error validating origin"？

这通常是开发环境问题：前端 dev server 端口（如 7001）与后端端口（如 8000）不一致，导致 WebAuthn origin 校验失败。生产环境中前后端同域名不会出现此问题。
