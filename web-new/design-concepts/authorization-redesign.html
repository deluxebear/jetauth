Design Concept

以业务系统为中心的
授权管理体验

将扁平的资源列表重构为以 Application 为维度的权限管理视图，让每个业务系统的权限配置一目了然

Design Problem

当前「授权」模块将 角色、权限、模型、适配器、执行器 拆成 5 个独立的扁平列表。对于需要集成权限管理的业务系统来说，管理员必须在多个页面之间跳转，手动记忆哪些角色属于哪个应用、哪条权限绑定了哪个模型。

核心矛盾：权限管理的对象是「业务系统」，但 UI 的组织维度是「资源类型」。这就像把一个公司的组织架构按「姓名、职位、部门、薪资」分成 4 个独立列表来管理——信息在，但关系断了。

Information Architecture

第一层：授权概览页 — 以卡片形式展示所有已绑定权限管理的应用，每张卡片显示角色数、权限数、模型、状态。管理员一眼看到全局。

第二层：应用授权详情页 — 点击任意应用卡片进入该应用的完整授权管理视图。Tab 页签切换：概况、角色、权限、测试、集成指南。所有数据都自动按该应用过滤。

模型/适配器/执行器不再作为独立的顶级菜单，而是下沉到每个应用的「概况」Tab 中作为配置项。高级用户仍可通过「全局设置」入口访问完整列表。

Key Design Decisions

设计决策

1. 侧边栏简化 — 「授权」下只保留一个入口「授权管理」，替代原来 5 个独立菜单。模型/适配器/执行器降级为「全局配置」的子页面，只有高级用户需要。

2. 应用卡片作为入口 — 每张卡片就是一个业务系统的「权限管理控制台」，显示关键指标（角色数、权限数、用户数、资源数），让管理员一眼看到全局健康度。

3. Tab 内自动过滤 — 进入某个应用后，角色列表、权限列表都自动按该应用过滤。新建角色/权限时自动绑定当前应用，减少操作步骤和出错概率。

4. 测试 Playground — 配置完权限后，最大的痛点是「不知道配对了没有」。内嵌的 Enforce 测试让管理员即时验证，无需切换到 Postman 或写脚本。

5. 集成指南自动生成 — SDK 代码自动填入当前应用的凭证，复制即可用。消除了「配置完权限但不知道业务系统怎么调用」的断层。

Implementation Notes

后端改动最小化 — 所有 API 接口不变。前端通过已有的过滤参数（owner、resourceType、model 等）实现按应用聚合显示。Enforce Playground 直接调用 /api/enforce。

向后兼容 — 「全局配置」入口保留完整的模型/适配器/执行器/角色/权限列表，高级用户和已有工作流不受影响。新 UI 是一个更高层次的聚合视图。

数据关联 — Permission 已有 Model 字段，Role 通过 Permission 间接绑定 Application（Permission.Resources 包含应用名）。前端聚合逻辑：查询该应用名下所有 Permission → 提取引用的 Role → 展示。
