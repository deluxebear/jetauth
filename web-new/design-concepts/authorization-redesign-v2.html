Screen 3 — 角色管理 + 用户抽屉
Key Interaction

用户数是可点击的链接，点击后打开右侧抽屉（Drawer），展示该角色下的所有用户。抽屉内支持搜索、分页、添加/移除用户。无论角色下有 3 个还是 5000 个用户都能正常工作。

角色列表只展示关键指标，不做行内展开——避免大量用户撑爆表格。「编辑」按钮跳转到完整编辑页处理继承关系、域等低频配置。

// 初始化 SDK — 已填入 ERP 系统凭证 casdoorsdk.InitConfig( "https://auth.company.com", // JetAuth Endpoint "a1b2c3d4e5f6", // ERP ClientId "secret_xxxxxxxxxxxxxxxx", // ERP ClientSecret "", "acme-corp", "erp-system", ) // API 中间件 — 每次请求校验权限 func AuthMiddleware(next http.Handler) http.Handler { return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) { userId := getUserFromJWT(r) allowed, _ := casdoorsdk.Enforce( "acme-corp/erp-read", "", "", "", "", casdoorsdk.CasbinRequest{userId, r.URL.Path, r.Method}, ) if !allowed { http.Error(w, "Forbidden", 403); return } next.ServeHTTP(w, r) }) }

Design Summary
Architecture

7 个页面，两个视角，一套数据

应用视角（授权管理）：概览 → 概况 / 角色 / 权限 / 测试 / 集成 — 回答「这个系统谁能做什么」

用户视角（用户编辑页）：权限概览 Tab — 回答「这个人在所有系统能做什么」

角色用户管理：点击用户数 → 右侧 Drawer（搜索 + 分页 + 添加/移除）— 3 人和 5000 人都能正常工作

后端零改动：所有现有 API 不变，前端通过已有过滤参数实现按应用聚合。「全局配置」入口保留完整的模型/适配器/执行器列表，向后兼容。
