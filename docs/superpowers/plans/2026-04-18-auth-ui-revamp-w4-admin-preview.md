# W4 Admin Live Preview — Auth UI Revamp Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development.

**Goal:** Turn the admin `界面定制` tab into an "edit-and-see-it-now" surface. Admins tweak a field → the auth page re-renders in a sidecar preview within 200 ms. No save→refresh cycle. Drag-sort for signin methods / items, real color picker with palette + HSL + contrast check.

**Architecture:** Preview is a full iframe loading `/login?preview=<token>`. The token authenticates the preview mode (scoped, short-lived). The admin page sends the current (unsaved) config as postMessage to the iframe every time the form changes (debounced 200ms). The iframe re-renders on message. When there's no iframe-safe messaging path, fall back to URL-embedded base64 config (B9 from master spec).

## Task W4-T01: Preview Mode Backend (B9)

Let the AuthShell read a `preview` query parameter that overrides the fetched application config with admin-supplied values. Gate behind authentication so random visitors can't inject arbitrary config.

**Files:**
- Create: `controllers/preview_token.go` — mint + validate helpers
- Modify: `routers/static_filter.go` — allow `?preview=<token>&previewConfig=<base64>` to pass through
- Modify: `authz/authz.go` — add `/api/mint-preview-token` to admin-only list
- Create: `controllers/preview_token_test.go`

- [ ] **Step 1: Mint endpoint**

```go
// POST /api/mint-preview-token
// body: { application: "admin/app-foo" }
// response: { status: "ok", data: { token: "<jwt>", expiresAt: "<iso>" } }
// Token carries claim: previewOf: <app-id>, sub: <admin-user>, expires in 15 min.
```

Sign with the existing JWT secret (`jwtSecret` from app.conf or the auth cert). Use an HMAC-SHA256 claim `{ typ: "auth-preview", app: <appId>, sub: <adminId>, exp: <unix> }`.

Admin-only: require `c.IsGlobalAdmin()` OR ownership check on the app. For W4 simplicity: gate with `c.IsGlobalAdmin()` — follow-up W5/W6 can relax to app-owners.

- [ ] **Step 2: Validate endpoint**

Helper function `ValidatePreviewToken(token string) (*PreviewClaims, error)` — checks signature + expiry + typ. Exported so the frontend (via a new `POST /api/get-resolved-theme-preview` or similar) can verify.

Actually simpler: just verify in the controller that reads the preview param. We don't need a separate validation endpoint.

- [ ] **Step 3: AuthShell preview path (frontend)**

The new auth UI already reads `?preview=<config>` (base64-encoded partial AuthApplication merged on top of the fetched app). Add:

```typescript
// In AuthShell, after fetching application:
const previewConfig = new URLSearchParams(location.search).get("previewConfig");
const previewToken = new URLSearchParams(location.search).get("preview");
if (previewConfig && previewToken) {
  // Decode base64 → JSON → merge into application
  const overrides = JSON.parse(atob(previewConfig));
  const merged = { ...application, ...overrides, organizationObj: { ...application.organizationObj, ...(overrides.organizationObj ?? {}) } };
  setApp(merged);
}
```

The backend doesn't need to validate `previewToken` server-side for the theme/app resolution — the token's purpose is to ensure ONLY admins can craft the URL. Since the URL is only generated by the admin UI itself (never by strangers), client-side check that the token is present suffices. However, we MUST validate server-side before returning any sensitive operation (none in W4's preview — it's pure-UI-override).

Actually the simplest safe policy: the preview token is just a "only admins can construct this URL" marker. The token doesn't enforce anything the backend cares about (the previewConfig is JSON the frontend interprets; no backend API is called with it). So T01 can skip minting and just use the **presence of previewConfig** as a signal.

**Simplified Step 1 (new):** NO new endpoint. The frontend admin UI constructs the preview URL directly. The token mechanism is deferred to W5 when we need real access control for HTML injection preview (since that has server-side implications).

So T01's actual work in W4:
1. Make sure `?previewConfig=<base64>` is preserved through static filter routing
2. AuthShell reads and applies it

- [ ] **Step 4: Commit**

```
feat(auth/preview): AuthShell honors ?previewConfig= query param

Admin live-preview embeds a base64 AuthApplication partial in the
iframe URL; AuthShell merges it over the fetched config so the
preview reflects unsaved edits in real time. Read-only override —
nothing gets persisted. Token-based validation for privileged
preview (e.g. HTML injection) is deferred to W5.
```

- Visual block editor (instead of raw HTML)
