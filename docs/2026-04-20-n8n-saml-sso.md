## 适用场景

把 JetAuth 作为 **SAML 2.0 IdP**，n8n 作为 **SP**，实现用户在 n8n 登录页点 "使用 SSO 继续"，跳到 JetAuth 完成认证后自动回 n8n，不再维护 n8n 独立的邮箱密码。

典型用途：公司内部用 n8n 做自动化工作流，账号体系收归 JetAuth 统一管理。

---

## 前置条件

- **n8n Enterprise 版**：SAML SSO 是 Enterprise 特性，社区版没有 SSO 菜单
- **JetAuth 管理员账号**：能编辑应用
- **一个 HTTPS 可达的 JetAuth 域名**：n8n 需要访问 JetAuth 的 `/api/saml/metadata` 拉元数据。本地开发用 Tailscale Funnel / Cloudflare Tunnel 都行；生产建议正式域名 + Let's Encrypt
- **n8n 也要 HTTPS 可达**：JetAuth 生成的 SAMLResponse 会 POST 到 n8n 的 ACS URL

---

## 整体流程

```
用户在 n8n 登录页 ──点击「使用 SSO 继续」──> n8n SAML SP
                                                │
                                                │ 生成 AuthnRequest，重定向到 IdP SSO URL
                                                ▼
                             浏览器跳到 JetAuth: /login/saml/authorize/admin/ERP
                                              ?SAMLRequest=...&RelayState=...
                                                │
                                                │ 用户输密码/扫码/走 OAuth provider
                                                ▼
                                  JetAuth 后端 /api/login (type=saml)
                                                │
                                                │ 返回 base64 SAMLResponse + SP ACS URL
                                                ▼
                              前端 samlPost.ts 建隐藏 form 自动 POST
                                                │
                                                ▼
                              n8n /rest/sso/saml/acs 验签 → 创建/登录用户
                                                │
                                                ▼
                                          n8n 工作台
```

## JetAuth 侧配置

全部在 **应用管理 → 编辑对应应用**（例 `jetems/ERP`）里完成。

### Step 1：确认应用归属与证书

**基本 标签页**：

- **组织**：用户实际登录的组织（例 `jetems`）
- **名称**：应用名（例 `ERP`）

注意 URL 里 `application=admin/ERP` 的 `admin` 是应用记录的**所有者**（Casdoor 约定），不是组织，不要混淆。

**安全设置 标签页**：

- **令牌证书**：选 `cert-built-in`（默认的 x509 + RSA 4096 证书）。SAML 签名用的就是这个，留空会报 `please set a cert for the application first`。

### Step 2：添加重定向 URI 白名单

**OAuth / OIDC 标签页** → **重定向 URI**：

添加 **一条** 条目：

```
https://n8n.jetems.com
```

**推荐只写域名**（不带协议+路径），后端会按域名匹配所有子路径（`object/application_util.go:402-415` 的无协议分支）。这样 n8n 的 Issuer（entity ID = `/rest/sso/saml/metadata`）和 ACS URL（`/rest/sso/saml/acs`）**一条覆盖两个**。

如果必须写完整 URL，需要同时添加两条：

```
https://n8n.jetems.com/rest/sso/saml/metadata
https://n8n.jetems.com/rest/sso/saml/acs
```

⚠️ 只加 `/acs` 不加 `/metadata` 会在 Issuer 校验阶段直接拒，报 `err: Issuer URI: ... doesn't exist in the allowed Redirect URI list`。

### Step 3：SAML 标签页设置

| 字段 | 推荐值 | 说明 |
|---|---|---|
| **SAML 回复 URL** | **留空** | 填了会**强制**覆盖 n8n AuthnRequest 里声明的 ACS URL。留空让 n8n 自己说了算 |
| **启用 SAML 压缩** | OFF | n8n 不要求压缩，关掉最兼容 |
| **启用 SAML C14N 1.0** | **ON** | n8n 用的 `@node-saml` 只认 exclusive c14n（`xml-exc-c14n#`），不开这个 JetAuth 默认用 `xml-c14n11`，n8n 直接拒 |
| **使用邮箱作为 SAML NameID** | 按需 | 开了之后 `<saml:NameID>` 是用户邮箱；关闭时是用户名 |
| **启用 SAML POST 绑定** | OFF | n8n **只支持 HTTP-Redirect binding** 的 SSO endpoint（`saml-validator.ts`）。元数据 URL 的 query 参数对应这个：`enablePostBinding=false` |
| **SAML 哈希算法** | **SHA256** | 默认 SHA1 n8n 拒。必须切 SHA256 |
| **禁用 SAML 属性** | OFF | 关闭后 Assertion 不带 AttributeStatement，n8n 就拿不到用户信息 |
| **启用 SAML 断言签名** | **ON** | n8n `wantAssertionsSigned=true`，Response 签名不够，必须 Assertion 本身也签 |

### Step 4：SAML 属性映射（关键）

n8n 按 **WS-Federation Claim URI** 查属性，Casdoor 原生的 `Email`/`Name`/`DisplayName` 它**不认**。必须在 **SAML 标签页 → SAML 属性** 里手动添加 4 条：

| 名称 (Name) | 名称格式 (NameFormat) | 值 (Value) |
|---|---|---|
| `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress` | Basic | `$user.email` |
| `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/upn` | Basic | `$user.email` |
| `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/firstname` | Basic | `$user.name` |
| `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/lastname` | Basic | `$user.name` |

**支持的占位符**（`object/user_util.go:1063-1067`）：
- `$user.email` / `$user.name` / `$user.id` / `$user.phone` / `$user.owner`
- `$user.roles` / `$user.permissions` / `$user.groups`（多值，每个值生成一个 `<saml:AttributeValue>`）

⚠️ **最容易踩的坑**：复制粘贴时 Name 字段前带了空格。n8n 精确比较 URI，首尾空格都会让它报 `missing attributes`。JetAuth 前端在保存时会自动 trim（`ApplicationEditPage.tsx:normalizeSamlAttributesForSave`），但建议粘贴后肉眼确认一下。

**用户要有 Email**：登录的账号 `email` 字段不能空，否则 n8n 收到空的 emailaddress 会拒绝。

## n8n 侧配置

n8n 管理后台 → **Settings → SSO** → **Select Authentication Protocol = SAML**。

| n8n 字段 | 填什么 |
|---|---|
| **重定向 URL (ACS)** | n8n 自己生成，只读；复制给 JetAuth 白名单参考 |
| **实体 ID** | n8n 自己生成（`https://n8n.jetems.com/rest/sso/saml/metadata`），只读 |
| **元数据 URL**（推荐这个，不用手贴 XML） | `https://你的JetAuth域名/api/saml/metadata?application=admin/ERP&enablePostBinding=false` |
| **用户角色自动开通** | 选 "在 n8n 中管理"（SAML 断言不带角色，新用户默认 member，手动改） |
| **已激活** | 打勾 |

点 **保存设置**，n8n 会立即 fetch 元数据并把 IdP 的公钥（从 cert-built-in 提取的 x509）存到自己的 DB 里。

### 元数据 URL 参数说明

`?application=admin/<appName>&enablePostBinding=<bool>`

- `application`：固定 `admin/` 前缀 + 应用名。**不是** `组织名/应用名`——Casdoor 里 Application 永远归 `admin` 组织所有者
- `enablePostBinding`：**必须 `false`**。n8n 只接受 HTTP-Redirect binding

## 验证

**点 n8n SSO 页的「测试设置」按钮**：

1. 浏览器跳到 JetAuth 登录页（URL 形如 `/login/saml/authorize/admin/ERP?SAMLRequest=...&RelayState=...`）
2. 输密码登录
3. 页面自动 POST 回 n8n ACS
4. n8n 显示测试成功

**如果要走真实 SSO 流程**：去 n8n 登录页，点 **使用 SSO 继续**，走一遍完整流程。首次登录的用户会在 n8n 里自动建账号。

## 常见错误排查

按踩坑时间顺序排列：

### 1. `canonicalization algorithm 'http://www.w3.org/2006/12/xml-c14n11' is not supported`

**原因**：JetAuth 默认用 `xml-c14n11` 签名，n8n 只认 exclusive c14n。

**修复**：SAML 标签页 → **启用 SAML C14N 1.0** 打开。

### 2. `ERR_FAILED_TO_VERIFY_SIGNATURE`

**原因**：签名算法/canonical 形式在两侧不一致。

**修复链**：
- **SAML 哈希算法** 选 **SHA256**（默认 SHA1 会挂）
- 确保 **启用 SAML 断言签名** 打开
- 让 n8n 重抓 metadata：SSO 设置页剪切元数据 URL → 保存 → 粘回去 → 再保存

### 3. `the signature value ... is incorrect`

**原因**：goxmldsig 签名时用了 `PrefixList="xs"` 但没把 `<ec:InclusiveNamespaces>` 写进 Transform，n8n 按空 PrefixList 重算摘要对不上。**已在 `object/saml_idp.go` 修复**（改成空 PrefixList + 移除 `xsi:type="xs:string"`）。

后端更新后重新编译重启 JetAuth：
```bash
go build -o /tmp/jetauth-bin . && pkill -f jetauth-bin; /tmp/jetauth-bin &
```

### 4. `Invalid SAML metadata` / `only SAML redirect binding is supported`

**原因**：n8n 只支持 HTTP-Redirect binding 的 SSO endpoint（`saml-validator.ts:validateIdentityProvider`），元数据里必须有 `<SingleSignOnService Binding="...HTTP-Redirect" .../>`。

**修复**：元数据 URL 的 `enablePostBinding=false`（这会让后端生成 HTTP-Redirect binding）。

### 5. `err: Issuer URI: https://n8n.jetems.com/rest/sso/saml/metadata doesn't exist in the allowed Redirect URI list`

**原因**：n8n 的 SP Entity ID 是 metadata URL，JetAuth 校验这个 URI 必须在应用的重定向 URI 白名单里。严格路径匹配。

**修复**：OAuth / OIDC 标签页的重定向 URI 加一条 `https://n8n.jetems.com`（不带路径，域名模糊匹配覆盖所有子路径）。

### 6. `Invalid SAML response (missing attributes: ...)`

**原因**：n8n 要的是 WS-Federation 标准 Claim URI，不是 Casdoor 原生属性名。

**修复**：按上面 [Step 4：SAML 属性映射](#step-4saml-属性映射关键) 配 4 条属性。

**还缺一个？**检查对应行的 Name 字段有没有首尾空格（复制粘贴最容易带）。

### 7. 登录成功后跳到 JetAuth 首页，不回 n8n

**原因**：前端 `web` 之前没写 SAML SP-initiated 的登录路由。

**状态**：已修。`/login/saml/authorize/:owner/:app` 现在在 `App.tsx` 注册（未登录用 `AuthShell`，已登录用 `SamlAuthorizePage`），登录成功后 `SigninPage.handleSamlResponse` 调 `submitSamlResponse` 自动 POST 到 SP ACS。

### 8. 通过第三方 provider（钉钉/GitHub/Apple）登录后不跳回 n8n

**原因**：AuthCallback 只完成 OAuth 登录就 redirect 到 `/`，丢了 SAML 上下文。

**状态**：已修。`AuthCallback.tsx` 会从 state 里 decode 出 `SAMLRequest` / `RelayState`，provider 登录完成后再调 `/api/login?type=saml` 换 SAMLResponse，自动 POST 到 n8n。

## 实现说明（面向开发者）

### 后端

核心文件：

- **`controllers/saml.go`**
  - `GetSamlMeta`：输出 IdP 元数据 XML
  - `HandleSamlRedirect`：SP 通过 HTTP-POST binding 推 SAMLRequest 到 `/api/saml/redirect/:owner/:app` 时的入口，303 重定向到前端登录页
- **`object/saml_idp.go`**
  - `GetSamlMeta`：组装元数据（根据 `enablePostBinding` 选 HTTP-POST 或 HTTP-Redirect binding）
  - `GetSamlResponse`：校验 AuthnRequest、签名 + canonicalization、base64 编码返回
  - `NewSamlResponse`：构造 SAML 2.0 Response XML 树（Issuer / Status / Assertion / AttributeStatement）
- **`controllers/auth.go:253-264`**：`/api/login` 在 `form.Type == "saml"` 分支调用 `GetSamlResponse`，把 base64 SAML Response 放在 `resp.Data`，SP 的 ACS URL 放在 `resp.Data2.redirectUrl`

### 前端

- **`web/src/auth/samlPost.ts`**
  - `submitSamlResponse`：建隐藏 `<form>` 自动 POST `SAMLResponse` + `RelayState` 到 SP ACS
  - `readSamlParams`：从 URLSearchParams 读 `samlRequest` / `SAMLRequest` 大小写两种 key
  - `LoginApiResponse` 类型
- **`web/src/auth/SamlAuthorizePage.tsx`**：已登录用户进入 SAML 授权页时，直接 POST `/api/login` 换 SAMLResponse 自动提交
- **`web/src/auth/signin/SigninPage.tsx` + `ClassicSigninPage.tsx`**：未登录用户的登录表单在识别到 `samlRequest` 时切换到 SAML 模式
- **`web/src/auth/AuthCallback.tsx`**：OAuth provider 回调后，如 state 里带 SAMLRequest 就继续完成 SAML 流程
- **`web/src/App.tsx`**：`/login/saml/authorize/:org/:app` 注册了两套路由（未登录 → AuthShell，已登录 → SamlAuthorizePage）

### 调试

前端有 debug 开关（**仅 DEV 构建生效**）：

```js
// 浏览器 Console
window.__JETAUTH_SAML_DEBUG = true
```

开启后 `submitSamlResponse` 不会真的 POST，而是把 base64 解码后的 SAML Response XML 打到 Console，并弹 alert 提醒。方便排查签名/属性问题。

## 附录：WS-Federation Claim URI 速查

n8n 以及大部分 Microsoft 系 SP 都用这套：

| Claim | URI |
|---|---|
| Email | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress` |
| UPN (User Principal Name) | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/upn` |
| First name | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/firstname` |
| Last name | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/lastname` |
| Name | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name` |
| Groups | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/groups` |
| Role | `http://schemas.microsoft.com/ws/2008/06/identity/claims/role` |

接入其他 SP（SAP、ADFS 系统、老 .NET 应用）通常也能直接用这套 URI。
