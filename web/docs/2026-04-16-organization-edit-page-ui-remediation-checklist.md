# OrganizationEditPage UI 整改清单

**日期**: 2026-04-16
**页面**: `web-new/src/pages/OrganizationEditPage.tsx`
**面向对象**: 前端研发
**用途**: 作为分阶段执行、排期和验收的整改任务清单

---

## 一、整改目标

本轮整改不追求大改视觉风格，重点解决以下问题：

- 提升页面的信息架构清晰度
- 修复配置能力与实际结果不闭环的问题
- 提升高风险配置的表达与防误操作能力
- 收敛页面内手写交互组件，降低维护成本
- 为后续设计系统化打基础

---

## 二、执行原则

- 优先修复“用户能感知到的错误”与“配置不可信问题”
- 优先补足结果反馈，而不是继续增加新配置项
- 不在本轮引入大规模视觉重设计
- 能抽基础组件的，不继续在页面内堆局部实现
- 每项整改都需要明确验收标准

---

## 三、P0 清单

### P0-1 修复菜单配置与真实路由不闭环

**问题**
- 菜单配置树来自 `navConfig.ts`
- 但部分路由在导航存在，在实际路由中不存在
- 用户配置后会出现“菜单能看到，但点击被兜底跳回首页”的结果

**涉及文件**
- `src/navConfig.ts`
- `src/App.tsx`

**执行项**
- 盘点 `navConfig.ts` 中所有 `to`
- 对照 `App.tsx` 中已注册路由
- 标记并处理以下情况：
  - 存在菜单，无实际路由
  - 存在路由，无菜单配置入口
  - 权限控制导致菜单可配但不可访问

**建议处理方式**
- 短期：先移除无实际路由的菜单项
- 中期：如业务必须保留，补占位页面并明确“建设中”

**验收标准**
- `navConfig.ts` 中每一个菜单项都能在前台找到明确归属
- 从组织菜单配置勾选出来的项，点击后不再回到首页兜底
- 不允许存在“可配置但不可达”的菜单项

---

### P0-2 修复当前页面相关的 build / type 问题

**问题**
- 当前工程存在 TypeScript 构建错误
- 与该页直接相关的问题会影响页面可信度与后续重构

**已知点**
- `OrganizationEditPage.tsx` 内 `SearchableSelect` 空结果使用了未定义的 `t`

**涉及文件**
- `src/pages/OrganizationEditPage.tsx`

**执行项**
- 修复该页直接相关的 TS 报错
- 修复该页内局部子组件的显式类型问题
- 避免继续扩大 `any` 的使用范围

**验收标准**
- 该页相关的 build 错误归零
- 该页新增代码不引入新的 TS 错误

---

### P0-3 为安全高风险配置增加风险提示和确认层

**问题**
- `Security` 标签页里高风险项和普通配置混在一起
- 风险级别、后果、推荐值都不明确

**重点字段**
- `masterPassword`
- `masterVerificationCode`
- `passwordObfuscatorType`
- `passwordObfuscatorKey`
- `ipWhitelist`
- `mfaItems`

**涉及文件**
- `src/pages/OrganizationEditPage.tsx`

**执行项**
- 按风险级别把 `Security` 内配置分组
- 为高风险项增加帮助文案
- 为会影响登录与组织接入的关键项增加确认操作
- 对明显非法值补前端校验

**建议最低实现**
- 新增 `Danger / Sensitive` 区块
- 使用 warning 风格提示
- 对关键修改增加 `modal.showConfirm`

**验收标准**
- 用户能一眼区分普通配置与危险配置
- 修改高风险配置时至少出现一次二次确认
- 输入明显非法值时前端有阻断或明确提示

---

### P0-4 补齐自定义下拉与树组件的基础交互行为

**问题**
- 当前自定义组件视觉可用，但基础行为不完整
- 页面越复杂，键盘和焦点问题越明显

**重点组件**
- `TreeCheckbox`
- `MultiSelectDropdown`
- `SearchableSelect`
- `TagsEditor`

**涉及文件**
- `src/components/TreeCheckbox.tsx`
- `src/pages/OrganizationEditPage.tsx`

**执行项**
- 统一支持 `Esc` 关闭
- 统一支持点击外部关闭
- 打开后聚焦到可交互输入
- 关闭后焦点返回触发器
- 统一键盘上下选择和回车确认策略

**验收标准**
- 不使用鼠标时也能完成主要配置流程
- 所有弹层打开与关闭行为一致
- 不再出现弹层开着但焦点丢失的问题

## 四、P1 清单

### P1-1 增加组织状态概览层

**目标**
- 在进入细节编辑前，先告诉用户“当前组织状态”

**建议内容**
- 默认应用是否配置
- 自定义主题是否启用
- MFA 是否配置完整
- LDAP 是否已接入
- 是否开放了组织管理员扩展编辑权限

**实现建议**
- 在当前 sticky header 下方增加一组 overview cards
- 先做只读摘要，不在首版引入复杂交互

**验收标准**
- 页面首屏无需滚动即可理解当前组织的大体状态
- 用户能快速判断应优先进入哪个标签页

---

### P1-2 重构 Theme 模块命名与边界

**问题**
- 当前 `Theme` 的可配置项较多，但实际影响范围主要集中在登录体验和品牌层
- 容易让用户误解为“整个后台主题系统”

**执行项**
- 根据当前能力边界重命名模块
- 优先推荐命名：
  - `Branding`
  - `Brand Theme`
  - `Login Experience`

**同步调整**
- 标签名
- section 标题
- 帮助文案
- 预览说明

**验收标准**
- 模块名称与真实能力范围一致
- 用户不会误以为 compact mode 已覆盖整个后台

---

### P1-3 为 Menu 配置增加结果预览

**问题**
- 现在只能配置树，不能直接看到结果

**执行项**
- 在 `Menu` 标签页内增加右侧预览区或下方预览区
- 支持查看：
  - Admin Nav 预览
  - User Nav 预览

**实现建议**
- 复用 `navConfig.ts`
- 做静态侧边栏渲染，不需要完整 router

**验收标准**
- 勾选菜单项后，预览能实时更新
- 用户能清楚知道配置结果长什么样

---

### P1-4 拆分 Advanced 标签页

**问题**
- `Advanced` 当前混合了系统集成和用户资料治理

**建议拆分方向**
- `Integrations`
- `Profile Schema`

**拆分建议**
- `Integrations`
  - LDAP attributes
  - LDAP servers
  - Kerberos
- `Profile Schema`
  - accountItems
  - 字段可见性/可编辑性相关能力

**验收标准**
- 单个标签页只服务一种明确任务心智
- 用户进入标签后能快速理解该标签的职责

---

## 五、P2 清单

### P2-1 为 Permissions 增加搜索与模板

**问题**
- 当前字段权限配置虽然完整，但仍偏底层

**执行项**
- 增加字段搜索
- 增加按场景预设模板

**推荐模板**
- 仅品牌配置
- 仅菜单配置
- 仅安全策略
- 仅用户资料治理

**验收标准**
- 用户不需要逐个 checkbox 才能完成常见授权配置

---

### P2-2 扩展组织主题能力到更多后台组件

**前提**
- 只有在产品确认要做“后台主题系统”时再推进

**执行项**
- 让 `compactMode` 影响：
  - 表单 vertical spacing
  - 表格行高
  - sticky header 高度
  - tabs 间距
  - section padding

**验收标准**
- 切换 compact mode 后，整个控制台密度发生一致变化
- 不再只影响登录页

---

### P2-3 抽离页面内局部交互件为统一基础组件

**目标**
- 降低页面内重复实现
- 为后续更多复杂页复用

**优先抽离组件**
- SearchableSelect
- MultiSelectDropdown
- TagsEditor
- 配置树控件

**建议目录**
- `src/components/primitives/`
- 或 `src/components/forms/`

**验收标准**
- 页面内不再保留多个相似但行为不完全一致的交互件实现
- 后续页面可直接复用

## 七、验收口径

本页整改完成后，应满足以下结果：

- 页面首屏能表达组织当前状态
- 菜单配置结果可预览、可达、可信
- 安全配置具有明确风险表达
- 页面内部复杂交互行为一致
- 标签结构比当前更符合任务心智
- 主题配置名称和能力边界一致

---

## 八、开发备注

- 本轮不建议直接大改整个页面视觉风格
- 本轮优先做结构修正和结果反馈闭环
- 如需重构，优先保持 `StickyEditHeader + Tabs + FormSection` 这套主骨架
- 设计系统层建议逐步收口，不建议一次性全面替换

---
